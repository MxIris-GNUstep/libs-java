4.8 Generating RPMs for the Wrapper Library

Please make sure you are confident about how the GNUstep make package manages generating RPMs in the standard case before you read this section.

4.8.1 Standard RPMs

The RPM for the wrapper library are generated independently from the RPM for the library itself. For example, suppose the library is packaged in a package called hippo. You have a hippo.spec.in file in the hippo project directory to generate the RPM for the hippo package. To generate an RPM for the wrapper, you need to add a 

 
hippo-wrapper.spec.in
file in the library directory, following the same syntax you use for the normal hippo.spec.in file. You can also add a 
hippo-wrapper.script.spec.in
file if you need. These two files are automatically copied into the JavaWrapper directory when JIGS generates it; JIGS uses these two files to set all up and ready to generate the RPM for the wrapper.

The RPM package for the wrappers will have name hippo-wrapper, and the same version number as hippo.

To create the RPM, you need to cd into the wrapper directory, and use the standard commands to build it (usually simply 

su
export RPM_TOPDIR=/usr/src/redhat
make rpm
(replace /usr/src/redhat with the appropriate directory for your system).

4.8.2 Debugging RPMs

When you create the debugging wrapper, the package is called hippo-wrapper-debug; and the files which are copied into the JavaWrapper_debug directory have different names: 

hippo-wrapper-debug.spec.in
hippo-wrapper-debug.script.spec.in

The debugging wrapper will not contain the Java classes, and it will depend on having installed the non debugging wrapper package first.

Hint on a counter-intuitive trick: make sure you type make rpm and not make debug=yes rpm inside JavaWrapper_debug when building the RPM.
